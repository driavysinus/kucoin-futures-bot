# Парадигма Фен-Шуй — торговая логика сопровождения позиций

## Ключевой принцип

Вся логика привязана к **размеру начального стопа** (расстояние от цены входа до SL). Это единица измерения для всех действий: порезов и трейлинга.

```
stop_size = abs(entry_price - sl_price)
```

---

## Поэтапная логика

### Исходные параметры при входе

```
entry_price  = цена входа
sl_price     = стоп-лосс (initial, далее не меняется до уровня 2)
tp_price     = entry + 3 * stop_size (для buy) / entry - 3 * stop_size (для sell)
stop_size    = abs(entry_price - sl_price)
contracts    = начальный объём в контрактах
```

Тейк-профит автоматически рассчитывается как **3 стопа** от входа (соотношение R:R = 1:3 на старте).

---

### Этап 1 — Прошли 1 стоп → Первый порез, SL не трогаем

**Условие:** цена достигла `entry ± 1 × stop_size`

**Действия:**
1. **Порез 50%** от начального объёма (округление вверх при нечётном)
   - Закрываем `ceil(contracts / 2)` контрактов по рынку
   - В рынке остаётся `contracts - ceil(contracts / 2)`
2. **SL остаётся на initial** (не двигаем)
3. **TP без изменений**

**Результат:** зафиксировали половину объёма в плюс, остаток ведём под прежним стопом.

```
Buy пример:  вход 0.6388, стоп 0.6368, stop_size=0.0020
             цена дошла до 0.6408 → порез 50%, SL и TP не меняем
```

---

### Этап 2 — Прошли 2 стопа → Второй порез + старт трейлинга

**Условие:** цена достигла `entry ± 2 × stop_size`

**Действия:**
1. **Порез 50%** от оставшегося объёма (округление вверх)
   - Закрываем `ceil(remaining / 2)` контрактов
2. **SL** переносим на `entry + 1 × stop_size` (buy) / `entry - 1 × stop_size` (sell) — это **стартовая точка трейлинга**
3. **TP без изменений** (углубление начнётся только с уровня 3+)

С этого момента **трейлинг-режим активен** — на каждом тике WebSocket `price_update` SL двигается за ценой.

```
Buy пример:  цена дошла до 0.6428
             Порез: ceil(остаток / 2)
             Новый SL = 0.6388 + 0.0020 = 0.6408
             TP = 0.6448 (без изменений)
```

---

### Этап 3+ — Трейлинг-SL + углубление TP (без порезов)

**Условие:** цена достигла `entry ± N × stop_size`, где N ≥ 3.

**Действия на каждом новом стопе (дискретно):**
- **TP** углубляем на `1 × stop_size` → `TP += stop_size` (buy) / `TP -= stop_size` (sell)
- Порезов больше нет.

**Трейлинг-SL (непрерывно, на каждом тике после 2-го пореза):**
- **buy:** `new_sl = price - stop_size`; если `new_sl > current_sl` → обновляем.
- **sell:** `new_sl = price + stop_size`; если `new_sl < current_sl` → обновляем.
- SL **двигается только в плюс**, никогда не откатывается.

```
Buy пример: stop_size = 0.0020, после 2-го пореза SL = 0.6408

цена 0.6438 → trailing new_sl = 0.6438 - 0.0020 = 0.6418 > 0.6408 → SL = 0.6418
цена 0.6450 → trailing new_sl = 0.6430 → SL = 0.6430
цена 0.6448 (уровень 3) → TP 0.6448 → 0.6468
цена 0.6430 (откат) → trailing new_sl = 0.6410 < 0.6430 → SL НЕ двигаем
цена 0.6428 → price <= 0.6430 → SL срабатывает, закрываем маркетом
```

---

## Сводная таблица

| Стопов пройдено | Порез | SL | TP |
|:-:|---|---|---|
| 1 | 50% от начального | initial (без изменений) | без изменений |
| 2 | 50% от остатка | entry ± 1×stop (старт трейлинга) | без изменений |
| 3 | — | трейлинг: `price ∓ stop_size` (динамически, только в плюс) | +1×stop |
| 4 | — | трейлинг | +1×stop |
| N | — | трейлинг | +1×stop |

Между уровнями SL обновляется на каждом тике WebSocket, а не только при пересечении дискретного уровня.

## Реализация — что хранится в Plan

```python
@dataclass
class Plan:
    symbol:         str
    side:           str        # buy / sell
    entry_price:    float      # цена входа (реальная, из позиции)
    contracts:      int        # начальный объём
    initial_sl:     float      # initial SL (для расчёта stop_size — не меняется)
    stop_size:      float      # abs(entry - initial_sl) — НЕ МЕНЯЕТСЯ
    leverage:       int

    # Текущее состояние
    remaining:      int = 0    # сколько контрактов осталось
    stops_passed:   int = 0    # 0 → 1 (1-й порез) → 2 (2-й порез) → 3+ (трейлинг)
    current_sl:     float      # текущий SL (трейлится после stops_passed >= 2)
    current_tp:     float      # текущий TP (углубляется с уровня 3+)

    entry_order_id: Optional[str] = None
    filled:         bool = False
    sl_triggered:   bool = False
    tp_triggered:   bool = False
```

## Мониторинг цены

Приходит WebSocket `price_update`. На каждом тике, в таком порядке:

```python
# 1. Уровневые события (дискретно): порез 1 / порез 2 / углубление TP
current_stops = int(distance / stop_size)
if current_stops > plan.stops_passed:
    for level in range(plan.stops_passed + 1, current_stops + 1):
        handle_stop_level(plan, level)

# 2. Динамический трейлинг SL (после 2-го пореза)
if plan.stops_passed >= 2:
    update_trailing_sl(plan, price)

# 3. Проверка SL / TP по обновлённым уровням
```

Порядок важен: уровни обрабатываются ДО проверки SL/TP, чтобы при пересечении нового стопа TP успел углубиться, иначе сработает старое значение TP.

Ордера SL и TP **виртуальные** — на бирже их нет. При касании — маркет-закрытие (`reduceOnly`) с последующей верификацией через `get_position` (до 3 повторов).
Активный `Plan` сохраняется в `plans.json` и после рестарта сверяется с реальной позицией на бирже; во время простоя процесса виртуальные уровни не исполняются.

## Совместимость

- Вход: `/open`, `/stop`, `/alert` — все используют одну логику.
- Параметры: `SYMBOL SIDE USDT PRICE SL [LEV]` — TRIG% не нужен.
- TP стартует на `entry ± 3 × stop_size`.
- Порезы: `ceil()` — всегда в большую сторону.
